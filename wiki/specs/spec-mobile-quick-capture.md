# Spec: Mobile Quick Capture Inbox

## 目标
让用户在移动端首页用最少步骤把临时想法快速记入当天 Inbox 文件，降低移动端捕捉想法的摩擦。

## 现状分析
当前移动端虽然支持在 Files tab 创建文件，但路径较长：Home → Files → 新建 → 输入文件名 → 进入编辑器。对于“先记下来再整理”的移动端场景，这个流程太慢，容易让用户放弃记录。现有 Home 页也没有为快速记录提供入口。

## 数据流 / 状态流
用户进入 Home 页
→ 点击 Quick Capture 卡片
→ Home 组件进入编辑态，展示当天 inbox 路径
→ 用户输入文本
→ 点击 Save to Inbox
→ 先检查 inbox 文件是否存在
→ 若存在：读取原内容并在末尾追加 capture 文本
→ 若不存在：创建新 inbox 文件并写入 capture 文本
→ 保存成功后清空输入，刷新首页 recent files
→ 当天 inbox 文件出现在最近活动中

## 方案
采用 Home 页内联 Quick Capture 卡片方案。

### User Flow

用户目标：在移动端打开应用后，用最少步骤把一个临时想法快速记入知识库。

前置条件：用户已连接 MindOS 服务，并已进入 Home tab。

Step 1: 用户在 Home 页看到 “Quick Capture” 卡片或按钮
  → 系统反馈：出现单行/多行输入框，聚焦键盘
  → 状态变化：进入 quick-capture 编辑态，默认目标为当天 Inbox 文件

Step 2: 用户输入一段内容
  → 系统反馈：保存按钮可点击；显示目标路径（如 `inbox/2026-04-11.md`）
  → 状态变化：本地维护 capture draft

Step 3: 用户点击 “Save to Inbox”
  → 系统反馈：按钮进入 loading；显示 “Saving...”
  → 状态变化：若目标文件存在则追加；不存在则自动创建并写入

Step 4: 保存成功
  → 系统反馈：显示成功提示，输入框清空；最近文件列表刷新，当天 Inbox 出现在最近活动中
  → 状态变化：draft 清空，文件树重新拉取

成功结果：用户的内容被追加到当天 Inbox 文件，且无需先创建文件名。

异常分支：
- 异常 A：网络错误 / 服务不可达 → 显示错误文案 + Retry，不清空输入内容
- 异常 B：用户提交空内容 → 按钮禁用，不发请求
- 异常 C：目标文件读取失败但保存可创建 → 直接走创建+写入
- 异常 D：保存 API 返回失败 → 显示失败原因，保留输入内容

边界场景：
- 内容只有空格 / 换行
- 内容很长（多段文本）
- 同一天多次连续 capture，应追加而不是覆盖
- 文件名含日期，跨天应自动切到新 inbox 文件
- 在保存中重复点提交

### 方案对比

方案 A：Home 页内联 Quick Capture 卡片
- 用户体验质量：⭐⭐⭐⭐⭐
- 实现复杂度：中
- 可维护性：高
- 风险：需要在首页拉通文件读写与刷新

方案 B：跳到 Files tab，再弹新建笔记
- 用户体验质量：⭐⭐
- 实现复杂度：低
- 可维护性：中
- 风险：路径长、打断思路

方案 C：单独新建 “Capture” tab
- 用户体验质量：⭐⭐⭐⭐
- 实现复杂度：中
- 可维护性：中
- 风险：标签栏拥挤，信息架构变重

选择：方案 A。
原因：最符合移动端“想到即记”的用户目标，不改导航结构，认知负担最小。

## 影响范围
- 变更文件列表
  - `mobile/app/(tabs)/index.tsx`
  - `mobile/lib/api-client.ts`（复用，不新增 API）
  - `mobile/lib/types.ts`（如需补充 UI 状态类型）
  - `mobile/lib/quick-capture.ts`（新增纯逻辑）
  - `mobile/__tests__/quick-capture.test.ts`（新增测试）
- 受影响的其他模块
  - Files tab 保持不变，仍承担正式文件管理职责
  - MarkdownEditor 不改动，Quick Capture 仅覆盖“快速记一条”场景
- 是否有破坏性变更
  - 无

## 边界 case 与风险
1. 输入为空：保存按钮禁用，避免空 note 污染 inbox
2. inbox 文件不存在：必须自动创建，不能要求用户先建文件夹/文件
3. 重复提交：saving 状态必须禁用按钮，避免重复追加
4. 风险：读-改-写存在并发覆盖窗口
   - mitigation：当前移动端是单用户个人知识库场景，先接受；后续可接入 append API
5. 风险：网络差导致用户以为没保存
   - mitigation：明确 loading、成功、失败三态，并保留失败草稿

## 验收标准
- [ ] Home 页出现明确的 Quick Capture 入口
- [ ] 用户无需输入文件名即可开始记录
- [ ] 默认保存目标是当天 inbox 文件（`inbox/YYYY-MM-DD.md`）
- [ ] 空内容时保存按钮禁用
- [ ] 保存中有明确 loading 反馈，且不可重复提交
- [ ] 目标文件不存在时会自动创建
- [ ] 同一天多次 capture 会追加到同一个 inbox 文件而不是覆盖
- [ ] 保存成功后显示成功反馈并刷新 recent files
- [ ] 保存失败时显示错误反馈，且输入内容不丢失
- [ ] 用户可以在成功后继续快速写下一条
