# Design Alternative Analysis: Tier 1 Features

## 1. TRASH / RECYCLE BIN — Alternative Designs

### 方案 A: Soft Delete to `.trash/` Directory

**描述**: 删除文件时将其移到 `.mindos/.trash/` 目录，保留原路径元数据，30天后自动清理。

```
UX质量：⭐⭐⭐⭐⭐ (完美)
实现复杂度：中等
可维护性：高
风险：文件移动需原子操作；git sync 需忽略 trash 目录

优点：
✓ 简单清晰，用户直观理解"回收站"概念
✓ 恢复是否成功都有明确反馈（吐司）
✓ 支持恢复冲突处理（覆盖/副本）
✓ 30天过期明确告知用户
✓ git 可以完全忽略 .trash/ 目录

缺点：
✗ 需要管理过期文件清理（定时任务）
✗ 如果用户删除大文件，磁盘占用24小时才清理

解决：启动时检查过期 + 每日后台任务
```

### 方案 B: 软标记 (Soft Flag) 不实际删除

**描述**: 文件保留原位置但标记为 `deleted: true` (YAML frontmatter)，UI 中隐藏，可恢复。

```
UX质量：⭐⭐ (差，用户不知道文件在哪)
实现复杂度：低
可维护性：中
风险：UI 混乱（隐藏的文件还在树里）；搜索结果混乱

缺点：
✗ 用户会困惑："我的文件去哪了？"
✗ 搜索时会返回已删除的文件，造成混乱
✗ 不符合用户预期（普通 app 的回收站）
✗ 30天后需要判断并清理，逻辑复杂

放弃此方案
```

### 方案 C: Git History 恢复（不做回收站）

**描述**: 依赖 git history 恢复，不实现专门的 Trash。用户通过 git log 找回。

```
UX质量：⭐ (太差，普通用户不会用 git)
实现复杂度：无需实现
可维护性：N/A
风险：用户体验差

缺点：
✗ 普通用户完全不会 git，无法恢复
✗ 需要开发者才能恢复，支持成本高
✗ 竞品都有回收站，用户会觉得 MindOS 不专业

放弃此方案
```

---

## 选择：**方案 A** ✅

**理由**:
- UX 最好（用户直观）
- 简单可靠（文件系统操作）
- 安全可预测（30天明确）
- 技术债少（无需复杂的元数据系统）

## 2. EXPORT — Alternative Designs

### 方案 A: 下载（使用 `Content-Disposition: attachment`）

**描述**: 后端将文件转换为目标格式，通过 `/api/file/export?path=...&format=html` 直接下载。

```
UX质量：⭐⭐⭐⭐⭐ (完美，一键下载)
实现复杂度：中等
可维护性：高
风险：大文件超时；浏览器下载限制

优点：
✓ 一键下载，最简单的 UX
✓ 用户立即得到文件，无需登录或云账户
✓ 支持进度条（stream response + Transfer-Encoding: chunked）
✓ 浏览器原生支持，无需额外库

缺点：
✗ 大文件（>500MB）可能超时
✗ 取决于服务器性能
✗ 用户网络断开会丢失下载进度

解决：分块流送 + 进度条 + 重试逻辑
```

### 方案 B: 云存储（上传到 S3/OSS）

**描述**: 后端将导出文件上传到外部云存储（AWS S3），返回下载链接给用户。

```
UX质量：⭐⭐⭐ (还不错，有链接)
实现复杂度：高（需要 cloud 配置）
可维护性：低（依赖第三方）
风险：泄露用户数据；额外费用

缺点：
✗ 用户隐私问题（文件上传到云）
✗ 需要额外配置（S3 credentials）
✗ 需要检查用户是否同意上传
✗ 额外费用（S3 存储 + 传输）
✗ 不符合 MindOS "local-first" 哲学

放弃此方案（除非用户主动要求）
```

### 方案 C: 预览 → 手动复制粘贴

**描述**: 只在浏览器中预览，用户手动复制文本。

```
UX质量：⭐ (太差)
实现复杂度：低
可维护性：低
风险：用户会放弃

缺点：
✗ 对大文件完全不可用
✗ 用户体验差（为什么不能一键下载？）

放弃此方案
```

---

## 选择：**方案 A** ✅

**理由**:
- UX 最好（一键下载）
- 本地优先（文件不上云）
- 用户完全掌控（隐私）
- 技术上可靠（流式 + 进度）

## 3. FAVORITES / PINNING — Alternative Designs

### 方案 A: localStorage 本地存储

**描述**: 使用 `localStorage['mindos-pinned-files']` 存储 PIN 列表。完全客户端。

```
UX质量：⭐⭐⭐⭐⭐ (完美)
实现复杂度：低（无 API 需要）
可维护性：高（纯前端）
风险：用户换设备不同步；localStorage 被清理丢失

优点：
✓ 响应极快（纯客户端）
✓ 无服务器开销
✓ 简单可靠，易调试
✓ 用户完全掌控（不同步意味着用户不被追踪）

缺点：
✗ 多设备之间不同步
✗ 用户清理浏览器数据会丢失 pins
✗ 不备份到 git（丢失可能性）

解决：
- 提供导出/导入 pins 的功能
- 提示用户在设置中备份 pins
- 用户可以手动同步到 .mindos/config
```

### 方案 B: Git 配置文件（.mindos/config/pinned-files.json）

**描述**: 将 PIN 列表保存到 `.mindos/config/pinned-files.json`，同步到 git。

```
UX质量：⭐⭐⭐⭐ (好)
实现复杂度：中等（需要文件同步）
可维护性：中等
风险：git 冲突；多人同时 pin 同一文件

优点：
✓ 跨设备同步（用户拉取 git 后自动恢复）
✓ 备份到 git（永久保存）
✓ 支持多人共享 pins（团队常用文件）

缺点：
✗ git 冲突可能（多人同时编辑）
✗ 文件 I/O 延迟（每次 pin 要写文件 + git）
✗ 复杂度高（文件同步 + 冲突处理）

折中：localStorage 作主，后台定期 sync 到 git（非关键路径）
```

### 方案 C: 数据库（SQLite）

**描述**: 使用本地 SQLite 数据库存储 PIN 表。

```
UX质量：⭐⭐⭐⭐ (好)
实现复杂度：高（需要后端支持）
可维护性：中（额外依赖）
风险：数据库损坏；过度设计

缺点：
✗ 过度工程化（PIN 列表只是数组）
✗ 需要 DB 迁移、备份
✗ 如果 DB 损坏，用户丢失数据
✗ 不适合小数据量

放弃此方案（太重）
```

---

## 选择：**方案 A（主）+ 方案 B（可选后续迭代）** ✅

**理由**:
- 方案 A UX 完美、实现简单
- 后续可以在设置中提供一键导出/导入功能
- v2 可选择加上方案 B 的 git 同步（非必需）
- 用户需要时可手动备份

## Pinned Files 顺序管理

### 方案 A: 拖拽重新排序

```
⭐ Pinned Files (3)
┌──────────────────────────────────┐
│ ≡ meeting-template.md            │ ← 拖拽手柄
│ ≡ 2026-Q1-goals.md               │
│ ≡ Team Standup Log.md            │
└──────────────────────────────────┘

UX: ⭐⭐⭐⭐⭐ (直觉，类似 Trello)
```

### 方案 B: 按钮排序（↑ ↓）

```
⭐ Pinned Files (3)
┌──────────────────────────────────┐
│ meeting-template.md  [↑] [↓]     │
│ 2026-Q1-goals.md     [↑] [↓]     │
│ Team Standup Log.md  [↑] [↓]     │
└──────────────────────────────────┘

UX: ⭐⭐ (丑陋，用户不想点按钮排序)
```

---

## 选择：**方案 A（拖拽）** ✅

**理由**:
- 更符合现代 UX（Gmail, Trello, Figma 都是拖拽）
- 用户直觉强
- 交互愉快感高

## Summary Table

| 功能 | 选择 | 理由 |
|------|------|------|
| Trash 实现 | 软删除到 `.trash/` | 清晰 + 简单 + 用户直觉 |
