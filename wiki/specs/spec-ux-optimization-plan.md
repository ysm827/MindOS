# MindOS 产品体验优化计划

*创建时间：2026-04-01 | 来源：全面 UX 审计 + 产品辩证分析*

---

## 诊断

**整体评分：7/10 — 功能齐全，指引不足。**

问题不在"缺什么"，而在"指路不清"：
- 核心功能都有了（文件 + AI Chat + Agents + Echo + Export + Trash）
- 用户**找不到**核心功能（"Echo" 名字不明白、"Discover" 被忽视）
- 空状态后又循环回新手引导，形成心理陷阱
- Agent 连接需要开终端 + 复制粘贴，破坏了 Web UX

---

## P0（<1 天）— 改变第一印象

### 1. 重命名模糊面板

| 当前 | 改为 | 理由 |
|------|------|------|
| Echo | 洞察 / Insights | "Echo" 不自解释，用户不会点 |
| Discover | 探索 / Browse | "Discover" 对非英语用户模糊 |
| Flows | 自动化 / Automations | "Flows" 太抽象 |

**为什么**：名字不自解释 = 功能不存在。这是 ROI 最高的改动。

### 2. 空状态从"循环"变"检查清单"

当前：选完模板 → 首页空 → 再看新手引导（"这是什么？"）

改为：选完模板 → 显示 3 步检查清单：
```
□ 写下你的第一条想法
□ 用 ⌘/ 和 AI 聊聊
□ 连接你的第一个 AI 工具（ChatGPT / Cursor）
```
每完成一个 ✅ 消失。满足感强，目标明确。

### 3. CSV / 文件操作添加保存反馈

当前：点保存 → 无反馈 → 用户不知道是否成功。

改为：Toast "已保存" / "保存失败，请重试"。一行代码，防止无效编辑。

### 4. Agent 连接：减少终端依赖

当前：Settings → Agents → MCP → 复制 token → 打开终端 → 粘贴（5 步 + 离开浏览器）

改为：Agent 面板内 → 选择 Agent → "一键配置"按钮 → 自动写入配置（或生成二维码/链接）

---

## P1（1 周）— 高频操作体验升级

### 5. Ask AI 上下文透明化

当前：⌘/ 打开 Ask → 用户不知道 AI 看的是当前文件还是全部笔记。

改为：Ask 面板顶部显示 "在与 {文件名} 交流" 或 "与全部笔记交流"（可切换）。

### 6. 搜索结果加上下文片段

当前：⌘K 搜索 → 只显示文件名 → 要点开才知道内容。

改为：搜索结果显示 "...{搜索词} 前后 10 字..." 片段，带高亮。

### 7. 文件头部加 Ask 按钮

当前：看完一个文件 → 想聊 → 要记住内容，然后 ⌘/ 再打字。

改为：文件顶栏右侧有 "Ask AI" 按钮 → 直接打开 Ask + 自动填入当前文件名。

### 8. AI 响应阶段可见化

当前：Ask AI → "Connecting..." → 等 → 回复（用户不知道在干嘛）。

改为：分阶段进度：
```
🔍 搜索相关笔记...
📖 阅读上下文...
🤔 思考中...
✍️ 生成回复...
```

## P2（立即实施）— 首页重构：单栏 + 系统脉搏条

### 设计决策：为什么不左右分栏

左右分栏（左 60% 文件列表 + 右 40% Agent Dashboard）有三个致命问题：
1. **视线分裂** — 用户打开首页第一意图是"继续工作"，不是看统计。两栏分散注意力
2. **移动端崩塌** — 双栏在手机上变上下堆叠，核心动作被推远或系统状态被忽视
3. **违背 Content is King** — 右栏 40% 展示"系统状态"是 chrome，不是 content

### 最终方案：单栏 + 内嵌脉搏条

保持单栏布局，在 Hero 区下方、文件列表上方，插入一条紧凑的**系统脉搏条**（System Pulse）。

### Section 精简

| 现状（5 section） | 目标（3 section） | 理由 |
|-------------------|-------------------|------|
| Pinned Files | **保留** | 用户主动标记，最重要 |
| Spaces | **保留** | 知识结构，核心概念 |
| Tools + Extensions | **合并到 Spaces 或移到 Discover** | 用户不关心内置/扩展区别 |
| Recently Edited | **保留，下移** | 高频但优先级低于 Pinned 和 Spaces |

### Phase 1 实施范围（立即做）

1. 新增 `SystemPulse` 组件 — 显示 Agent 连接状态（数据来自已有 `useMcpData`）
2. 精简 HomeContent sections — 去掉 Tools + Extensions，只保留 Pinned → Spaces → Recent
3. 脉搏条自适应 — 0 Agent / 1 Agent / 3+ Agent 三种状态
4. 脉搏条可折叠 — localStorage 记住折叠状态

核心原则：**不是"你是谁"决定看到什么，而是"你做了什么"决定看到什么。**

## 产品形态升级：从"笔记软件"到"人机共治控制台"

*2026-04-01 战略讨论*

### 核心洞察

MindOS 的护城河不是 Markdown 编辑器（Obsidian/Notion/Typora 已打完这个赛道），而是**同时连接多个 AI Agent 的知识中枢**。如果首屏还在强调"写笔记"，就是在用最弱的维度跟别人最强的维度竞争。

"Agent 读取你的知识库 47 次"这个数字如果显示在首屏，会产生**不可替代感**：用户意识到"我的 AI 们一直在用我的笔记，停用 MindOS 它们就变笨了"。这是 Notion 做不到的。

### 辩证分析

**说对了：**
- 定位层面完全正确——"人机共治的控制台"比"AI 笔记本"高一个维度
- 用户身份从"我在写东西"升级为"我在管理一个 AI 团队"，粘性和付费意愿更强

**说得太极端了：**
1. "首屏绝对不应该是写字的地方"——用户打开 MindOS 的第一需求仍然是快速记一个想法。Dashboard 是"回来看"的东西，不是"每次打开都需要"的。Slack 首屏是消息列表（核心动作），不是统计面板
2. "47 次读取"——需要真实数据支撑。如果用户只连了 1 个 Agent 一天用 3 次，"今日读取 3 次"反而暴露产品空虚。数字只有足够大时才有冲击力
3. "Markdown 只是底层协议"——低估了写作本身的价值。写作是思考的过程，很多用户的核心需求是"写"不是"管 Agent"

### 结论：不是二选一，是两个维度共存

**左手写字，右手管 Agent。**

首屏左侧是"快速行动"（最近文件 + 新建 + Ask），保证核心循环零摩擦。右侧是"神经中枢"（Agent 状态 + 统计 + 洞察），让用户感知系统在运转。

既不损害"写"的效率，又展示"控制台"的价值。

### 设计原则

1. **左手写字，右手管 Agent** — 核心循环零摩擦，系统状态被动展示不干扰
2. **数字只在有意义时展示**：
   - 0 Agent → 不显示统计，改为引导"连接你的第一个 AI"
   - 1 Agent 且读取 <5 次 → "多用 AI 聊聊，它会越来越懂你"
   - 3+ Agent 且读取 >20 次 → 完整统计面板（此时数字有冲击力）
3. **洞察是"行动建议"，不是"统计报告"** — "Agent 读取 47 次"只是数据；"你频繁提到安全性但没有规则，要添加吗？"是行动建议。后者才让用户觉得"这个产品在帮我变强"
4. **像飞行员仪表盘** — 不妨碍你开飞机，但随时告诉你系统状态

### 实现路径

| 阶段 | 做什么 | 效果 | 前置条件 |
|------|--------|------|---------|
| **Phase 1** | 首页右侧加 Agent 状态卡片（连接/活跃/空闲） | 用户感知"系统在运转" | MCP 连接状态 API 已有 |
| **Phase 2** | 加读取统计 + 最新洞察推荐 | 用户感知"AI 在学习" | 需要 Agent 读取计数 API |
| **Phase 3** | 加行动建议（基于 Echo 数据） | 用户感知"AI 在帮我变强" | 需要 Echo 推荐引擎 |

Phase 1 可以在现有首页上改，不需要重写。

### MindOS 的终极形态

不是"Dashboard 替代编辑器"，而是**"编辑器旁边永远有一个活的 Agent 状态面板"**——像飞行员的仪表盘，不妨碍你开飞机，但随时告诉你系统状态。

用户的心智模型从"我在用一个笔记软件"变成"我在运行一个人机共治系统，笔记是我和 AI 的共享语言"。
