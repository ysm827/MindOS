# 技术支柱 (Technical Pillars)

MindOS 的长期技术壁垒不在于"功能多"，而在于定义 Agent 时代操作系统的核心组件。

## 为什么是这四个

从 MindOS 的三大痛点（记忆割裂、记忆黑箱、经验断流）出发，解决路径上必然遇到四个系统级问题：

1. 多 Agent 并发读写无序 → **群体调度**
2. 协作经验随对话消失，无法自动沉淀 → **经验编译**
3. 知识库长期使用后质量退化 → **记忆代谢**
4. 通用 Agent 不懂个体用户 → **认知镜像**

"安全与隐私"是贯穿所有 Pillar 的横切关注点（本地优先、路径沙箱、写保护已内嵌于架构），不单独成柱。

## 结构关系

**知识生命周期的四个阶段：环境 → 产生 → 维护 → 涌现。**

```
Pillar 1 (群体调度)  ──→  Pillar 2 (经验编译)  ──→  Pillar 3 (记忆代谢)  ──→  Pillar 4 (认知镜像)
   环境：谁能进来           产生：经验变知识           维护：知识不腐化           涌现：深度理解用户
```

类比操作系统启动顺序：内核先起来（进程管理、权限、并发控制）→ 编译器可用（源码→可执行产物）→ 文件系统跑起来（存储、清理、索引）→ 用户态应用（个性化体验）。

工程上四条线可并行推进，但产品逻辑上存在前置依赖：没有有序的多 Agent 环境，经验编译和记忆代谢的数据来源不可靠；没有前三者的积累，认知镜像无从涌现。

## Pillar 1 — 异构群体智能调度与治理

**类比：** MindOS 的 OS 内核——不只提供数据，更负责多 Agent 的进程管理。

**核心问题：** 用户同时接入 5-10 个异构 Agent（写代码的、查资料的、管日程的），它们并发读写同一个知识库，谁来维持秩序？

**起点：** MCP Server 已支持多 Agent 同时连接（HTTP + stdio），Bearer Token 认证提供身份识别。用户已在同时使用 Claude Code + Cursor，并发问题现实存在。详见 [系统架构 → MCP](./20-system-architecture.md)。

**研究方向：**

| 方向 | 命题 | 预期突破 |
|------|------|---------|
| 任务感知的上下文路由 | 根据 Agent 当前任务类型和关键性，动态裁剪返回的上下文——重构代码的拿完整架构文档，拼写检查的只拿当前段落 | Agent 无需自己拼 prompt，系统级上下文优化 |
| 并发冲突解决 | 两个 Agent 同时修改同一文件时的锁机制、乐观并发控制、冲突合并策略 | 多 Agent 并发写入不再丢失数据 |
| 细粒度权限 | "Claude Code 可写代码区但只读 Profile""Cursor 可读架构文档但不能碰 SOP" | 超越读/写二分的语义级访问控制 |

**应用层扩展：**
- **多 Skill 编排** — 多个 Skill 的执行逻辑如何组合、编排、检测冲突；多 Agent 协作完成复合任务时的消息传递和结果聚合。建立在上述三个基础设施能力之上。

**与 Roadmap 的对应：** P3（Agent Governance、多 Agent 协作中枢、ACP 协议）。并发冲突解决的基础工程可在 P2 提前启动。

## Pillar 2 — 经验编译与增量演化

**类比：** MindOS 的经验提炼器——从运行时交互中萃取可复用的知识。

**核心问题：** LLM 已经能读懂自然语言 SOP 并执行。真正的瓶颈不是"Agent 读不懂指令"，而是**执行经验无法自动回流为可维护的 SOP**——用户纠正了 Agent 的错误，这个纠正只活在当前对话里，下次新对话又从零开始。

**起点：** Skills 的 SKILL.md + experience.md 已形成半结构化的行为定义和人工经验回流机制，但 experience.md 完全依赖人工编写，系统没有自动从交互中提取和结构化的能力。详见 [系统架构 → Skills](./20-system-architecture.md)。

**研究方向：**

| 方向 | 命题 | 预期突破 |
|------|------|---------|
| 反向编译（主线） | Agent 执行出错 → 人类修正 → 系统自动提取交互增量 diff → 结构化为 SOP 补丁 → 反映回 Markdown 供审计。处理的是**实时交互增量**（对话→SOP），区别于 Pillar 3 的静态文件去重 | 人机协作经验零损耗沉淀，SOP 自动演化 |
| 增量演化 | 在已有 SOP 上做局部修改（插入分支、修改条件、归档旧路径），而非全量重写 | 知识结构持续迭代而不破坏已有验证 |
| 正向编译（待验证） | LLM 直读 Markdown 在多步条件分支、跨文件依赖、执行顺序敏感场景下是否存在系统性缺陷？ | 如果 bad case 成立，引入中间表示（行为树 / DAG）提升执行一致性；如果不成立，说明 prompt 注入已够用，正向编译无需投入 |

**与 Roadmap 的对应：** 独立研究线。P2 的 Skills 增强是前置工程准备，核心算法落地预计 P3+。

## Pillar 3 — 动态记忆代谢与自组织心智网络

**类比：** MindOS 的文件系统——让知识库从"死水"变成有生命周期的活体。

**核心问题：** 知识库只增不减，Agent 越用越迟钝。RAG 检索出一堆过期的、矛盾的、重复的内容（上下文污染）。

**起点：** Wiki Graph 提供文件间链接拓扑，Agent Inspector 记录工具调用日志（"谁在用什么"）。Knowledge Health 的过期检测、矛盾扫描已纳入 P2 规划。详见 [Roadmap → P2](./01-project-roadmap.md)。

**研究方向：**

| 方向 | 命题 | 预期突破 |
|------|------|---------|
| 记忆分层 | 工作记忆（对话上下文）→ 短期记忆（近期交互沉淀）→ 长期记忆（全局 SOP / Profile），三层之间的升维和降级机制 | 不同时效的知识自动归位，减少上下文污染 |
| 活跃度感知 | 基于调用频率、编辑频率、引用关系，维护每个知识节点的"温度" | 高温节点预热加速检索，低温节点自动归档 |
| 碎片折叠 | **已有静态文件**中多条记录语义高度重叠时，自动合并提炼为更高层级的抽象节点。区别于 Pillar 2 的反向编译（处理实时交互增量） | 知识库自动"瘦身"，信噪比持续提升 |
| 矛盾检测与解决 | 同一主题的不同文件出现矛盾描述时，主动标记冲突并引导用户裁决 | 消除隐性矛盾，保证知识库一致性 |

**与 Roadmap 的对应：** P2（Knowledge Health 是工程前置）、P3（深度 RAG、Agent Memory Layer）

## Pillar 4 — 个性化认知镜像与隐性意图对齐

**类比：** MindOS 的用户态——工具可以被复制，但"懂你"的程度无法被复制。

**核心问题：** 通用 Agent 对所有用户一视同仁。MindOS 掌握用户最深度的交互数据，如何把这种理解转化为不可替代的体验？

**起点：** Profile 目录存储用户身份/偏好/目标/风格，Agent Inspector 记录交互轨迹，experience.md 可作为偏好推断的原始数据源。详见 [系统架构 → Profile](./20-system-architecture.md)。

**与通用推荐系统的关键区别：** MindOS 的数据源不是点击流或浏览行为，而是**纯文本交互日志 + 用户自主编写的知识库**。本地优先意味着没有大规模协同过滤的数据基础，必须从单用户的长期文本轨迹中推断偏好——这本身就是一个独特的研究约束。

**研究方向：**

| 方向 | 命题 | 预期突破 |
|------|------|---------|
| 文本轨迹偏好推断 | 从工具调用日志（频繁使用/采纳/拒绝）和编辑行为（修改 Agent 写入的内容）中提取隐性偏好 | Agent 不问就知道用户风格，减少重复沟通 |
| 心智预设注入 | 外部 Agent 通过 MCP 请求上下文时，响应附带结构化用户特征摘要（`_meta.user_profile`），Agent 可选择性注入 system prompt。初期从 Profile 静态生成，后期从交互轨迹动态更新 | 任何新接入的 Agent 立刻"认识"用户 |
| 意图预测 | 基于当前工作上下文和历史模式，主动推荐可能需要的知识节点 | 从被动搜索变为主动推送，减少认知负载 |
| 演化追踪 | 记录用户认知模型随时间的变化轨迹（Mind Diff） | 用户能回溯"三个月前我是怎么想这个问题的" |

**与 Roadmap 的对应：** P4（长期探索）。数据采集基础可在 P2-P3 的 Agent Inspector 增强中积累。
