🔴 P0 关键缺陷
────────────────────────────────────────────────────────────────────────

1️⃣  ACP 代理路由无应用层重试
   ├─ 问题：ask/route.ts:664 调用 promptStream() 时无 try-catch-retry
   ├─ 影响：ACP 流式传输任何网络中断 → 立即失败
   ├─ 方案：包装 promptStream 用同样的重试循环（MAX_RETRIES=3）
   └─ 估计工作量：30-40 分钟

2️⃣  指数退避工具被闲置不用
   ├─ 定义位置：lib/agent/reconnect.ts (retryDelay, sleep)
   ├─ 使用位置：从未被导入调用
   ├─ 硬编码替代：ask/route.ts:758 使用 Math.pow(2, attempt-1)
   ├─ 问题：违反 DRY 原则，易漂移
   └─ 方案：统一改用 reconnect.ts 的工具函数

3️⃣  重试循环逻辑bug：最多只重试 2 次而非 3 次
   ├─ 代码：ask/route.ts:755 条件 `attempt < MAX_RETRIES`
   ├─ 循环：attempt=1,2 时满足 → 真正重试 2 次
   │       attempt=3 时 canRetry=false → 不执行循环体
   ├─ 结果：3 次重试设定但实际只有 2 次
   └─ 方案：改为 `attempt <= MAX_RETRIES` 或调整循环结构

4️⃣  转瞬性错误检测不完整
   ├─ 当前：retry.ts 仅基于 Error.message 文本匹配
   ├─ 缺失：HTTP 状态码参数，无 Retry-After 头支持
   ├─ 风险：DNS 错误、SSL 错误、Connection reset 可能漏判
   └─ 方案：增加 httpStatus 参数，支持直接状态码检查


🟡 P1 重要问题
────────────────────────────────────────────────────────────────────────

5️⃣  流式传输中断无恢复
   ├─ 问题：SSE 流中断后只能标记工具为 error，无重连
   ├─ 现象：长时间任务易因网络波动而失败
   ├─ 原因：stream-consumer.ts 依赖 reader.read() 循环，中断即终止
   └─ 改进：实现客户端断线重连或心跳检测

6️⃣  超时配置离散化无统一管理
   ├─ apiFetch: 30s
   ├─ sendAndWait: 30s (RPC)
   ├─ registry: 10s
   ├─ session.prompt: 由 pi-ai 决定（60-300s）
   └─ 缺失：无环境变量、无配置文件、无动态调整

7️⃣  SSE 流无心跳保活
   ├─ 问题：TCP keep-alive 间隔通常 > 2 分钟
   ├─ 风险：中间件（代理、NAT、负载均衡）可能切断长连接
   └─ 方案：定期发送 :heartbeat 或 :keep-alive SSE 事件


⚪ P2 可选改进
────────────────────────────────────────────────────────────────────────

8️⃣  请求去重防护
   ├─ 问题：重试时可能导致同一 tool 调用执行多次
   ├─ 方案：追踪已发送的 tool 调用 ID

9️⃣  可观测性不足
   ├─ 缺失：详细重试日志、失败分类统计
   └─ 改进：structured logging for retry events

┌─────────────────────────────────────────────────────────────────────────┐
│ 📈 性能影响预估                                                          │
└─────────────────────────────────────────────────────────────────────────┘

场景               总延迟          最坏情况
─────────────────────────────────────────
首次成功           0s              30-60s (timeout)
1 次重试成功       1s              1s + 60s = 61s
2 次重试成功       3s (1+2)        3s + 60s = 63s
全部失败           3s (1+2)        3s + 90s = 93s (3×30s)

备注：
- 单次 LLM 调用超时由 pi-ai 决定（通常 30-60s）
- 重试仅在"流未开始"时触发
- ACP 路由当前无延迟，但也无保护（直接失败）
