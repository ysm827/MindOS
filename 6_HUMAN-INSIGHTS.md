# Human Insights — Agent 协作经验

> 与 AI Agent 协作中积累的真实经验。每条 insight 来自实际踩坑，不是理论推导。

---

## 一、核心认知转变

### 你是产品经理，不是调度器
```
之前：开 3 个 terminal → 手动分配任务 → 逐个切换看状态 → 逐个催 review
之后：发一条指令描述终态 → Agent 全自驱 → 你只做最终验收
```
人的时间应该花在"定义做什么"和"判断做得对不对"，中间的执行、review、测试、提交全部交给 Agent。

### 质量靠系统保证，不靠人盯
三层防线，到位后人工 review 变成抽检而非必检：
- **CLAUDE.md 自检清单** = 编译期检查（每次改动自动执行）
- **wiki/80-known-pitfalls.md** = 已知 bug 数据库（Agent 改动前查阅）
- **dep-safety.test.ts 类防护测试** = 运行时 guard（CI 自动拦截）

## 二、指令设计

### 描述终态，不描述步骤
```
❌  "修这个 bug" → "review 下" → "再 review" → "commit" → "push" → "release"
✅  "修这个 bug，自检通过后 commit push release patch"
```
一轮 vs 六轮。Agent 擅长串联步骤，人擅长定义终态。

### 把隐含的质量标准说出来
Agent 不知道你心里的"完成"标准：
- "跑完测试" → 不用你说 review
- "更新 wiki" → 不用你事后补
- "release patch" → 不用你追加指令
- "完整闭环" → Agent 自动走 修复 → 全局扫描 → 补测试 → 记录 pitfall → 更新 backlog

### 并行任务在一条消息里给出
```
❌  开 3 个 terminal 分别执行，自己做调度器
✅  一条消息列出 3 个独立任务，说明"并行执行"
```
Agent 用 sub-agent / Teams 并行处理，你只看一个窗口。

### Bug 报告附上下文
```
❌  "sidebar 不更新"
✅  "sidebar 不更新。创建文件后文件树没有刷新。在 dev 模式复现。"
```
多一句上下文，少两轮排查。

## 三、Review 效率

### "Review 下"是最大的效率杀手
每次人工要求 review 浪费一轮交互。正确做法：
1. CLAUDE.md 写好自检清单，Agent 改完自动执行
2. 你只在最终结果出来后做一次验收
3. 不满意时给方向性反馈，不给泛泛的"review 下"

### Review 给方向，不给指令
```
❌  "你 review 下"（没有重点，Agent 泛泛检查，大概率漏掉关键点）
✅  "缓存失效路径完整吗？Next.js client/server 两层都覆盖了吗？"
```

### 不确定时一次问清
```
❌  Agent 猜一个方案 → 你说不对 → 再猜 → 又不对
✅  Agent 列出 2-3 个选项 + 推荐 → 你选一个 → 一次到位
```

## 四、防止返工

### 框架陷阱提前文档化
踩过的坑写入 `wiki/80-known-pitfalls.md`，Agent 每次改动前查阅：
- `force-dynamic` 只对 page/route 有效，layout 上无效
- `revalidatePath` 在测试环境会抛异常，需要 try-catch
- `--prefer-offline` 没有 fallback 会在新机器崩溃
- npm semver `^` range 可能引入不存在的 subpath export

### 让 Agent 写防护测试
每个 class of bug 对应一个自动化测试，不靠人记住：
- 版本范围 vs 实际 import → `dep-safety.test.ts`
- 裸 `--prefer-offline` 使用 → lint test 自动扫描 bin/

### 依赖关系决定串并行
- 改不同文件的独立 bug → 并行
- 同一文件的多个改动 → 串行
- 调研 + 实现 → 先调研后实现（但调研可以多路并行）

## 五、测试策略

### Agent 最容易跳过的就是测试
没有明确要求时，Agent 倾向于"写完代码跑一下现有测试通过就行"。实际上大部分 bug 都藏在未覆盖的路径里。

**解法**：在 CLAUDE.md 里写死"测试先行 + 三类 case 必须覆盖"，让 Agent 没有跳过的空间。

### 三类 case = 最小完备测试集
```
正常路径：用户按预期操作会怎样？
边界 case：输入为空 / 极大 / 极小 / 特殊字符会怎样？
错误路径：外部服务挂了 / 文件不存在 / 权限不够会怎样？
```

这三类不是理论——回顾项目踩过的坑：
- `revalidatePath` 在测试环境抛异常 → **错误路径**未覆盖
- `--prefer-offline` 在空缓存机器崩溃 → **边界 case**（首次运行）
- SDK `^1.6.1` 允许了不存在的 subpath → **边界 case**（版本下界）

每个都可以通过"写测试时过一遍边界清单"提前发现。

### 测试先行 vs 事后补测试
```
测试先行：写测试 → 红灯 → 写实现 → 绿灯 → 重构
事后补测试：写实现 → "应该没问题" → 上线 → 出 bug → 补测试

区别：前者把 bug 消灭在写代码时，后者等用户发现了再修。
```

对 Agent 来说测试先行还有个额外好处：测试本身就是 spec 的可执行验证。Agent 写完测试后，实现阶段的方向就被约束住了，不会跑偏。

### Bug fix 必须附带复现测试
```
❌  修了 bug → 手动验证 → 提交（下次同类 bug 再来一遍）
✅  写一个能复现 bug 的测试（红灯）→ 修复 → 测试变绿 → 提交（永久防护）
```

## 六、代码质量

### Agent 自我 review 容易泛泛——给它 checklist
不加约束的"review 一下"会变成：
- "代码看起来不错"
- "没有明显问题"
- "建议加一些注释"

**解法**：在 CLAUDE.md 里定义结构化 checklist，按 4 个维度逐条过：

| 维度 | 检查什么 | 为什么重要 |
|------|---------|-----------|
| **正确性** | 对照 spec 验收 + 查已知陷阱 | 做的是不是对的 |
| **健壮性** | 错误处理 / 输入验证 / 超时 / 竞态 | 异常情况下会不会崩 |
| **可维护性** | 死代码 / 重复代码 / 命名 / 注释 why | 下次改动会不会踩坑 |
| **性能** | N+1 查询 / OOM 风险 / 缓存三层 | 数据量大了会不会卡 |

### Code review 从 2 轮升级为 3 轮
```
轮 1：正确性 — 做的对不对
轮 2：健壮性 — 异常扛不扛得住
轮 3：可维护性 + 性能 — 未来好不好改、快不快
```

每一轮有明确的 checklist，不是"再看一遍"。

### 质量不是最后一步做的事
```
❌  写完所有代码 → 最后统一 review → 发现设计问题 → 大面积返工
✅  测试先行约束方向 → 每个模块写完立即 review → 问题局部修复
```

越早发现问题，修复成本越低。测试先行 + 逐步 review 比"最后大检查"高效 10 倍。

## 七、Spec 模板化

### 自由文本 Spec 是 bug 的温床
这次 sidebar 不更新的根因是三层缓存叠加（客户端 router cache / 服务端 revalidate / 内存 cache）。如果 spec 阶段有一段"数据流图"，要求画出"谁读 / 谁写 / 中间经过几层缓存"，这个问题在写代码之前就能暴露。

**解法**：定义 spec 模板，每段都不能为空。关键段落：
- **数据流 / 状态流** — 画出改动涉及的读写路径和缓存层。这是 self-review 时最重要的锚点。
- **边界 case 与风险** — 至少列 3 个。逼 Agent 在写代码前就思考异常路径。
- **验收标准** — 可执行的 checklist，不是"应该能工作"，而是"点击创建文件后，sidebar 在 1s 内显示新文件"。

### Spec self-review 有了锚点
```
之前：Agent 看一遍 spec，"看起来没问题"
之后：Agent 对照模板逐段检查，数据流图画了吗？边界 case 列了吗？验收标准可执行吗？
```
模板把"泛泛 review"变成"逐项验证"。

## 八、视觉回归验证

### vitest 通过 ≠ UI 正确
单元测试检查的是逻辑，不是视觉。这次 MCP config snippet 的 UI 改动，vitest 全通过，但你还是要求截图才放心——因为 CSS 布局问题、间距错误、组件遮挡这些单元测试完全覆盖不到。

**解法**：UI 改动时，code review 通过后、commit 前，用 Playwright 截图关键页面：
```
改了组件 → 跑测试通过 → Playwright 截图 → 附到最终呈现结果 → 你一眼验收
```

这样你的验收从"打开浏览器手动翻页面"变成"看几张截图"，时间从分钟级降到秒级。

### 什么时候需要，什么时候不需要
- 改了 TSX / CSS / 布局 → 需要
- 纯后端 / API / 配置 / 文档 → 不需要

过度截图是浪费，关键是**改了什么就截什么**。

## 九、发版后冒烟验证

### "本地没问题"是最危险的假设
两个真实案例：
1. `--prefer-offline` 本地跑没问题（有缓存），新机器 `npx` 安装直接崩（无缓存）
2. SDK `^1.6.1` 本地跑没问题（lockfile 锁了 1.27.1），新安装解析到 1.18.1 缺 express.js

这两个 bug 的共同特征：**开发环境永远无法复现，只在全新环境才暴露**。

**解法**：release 后在临时目录模拟全新安装：
```bash
cd /tmp && mkdir smoke-test && cd smoke-test
npx @geminilight/mindos@latest --version   # 确认版本号
npx @geminilight/mindos@latest --help       # 确认能启动
```

成本极低（30 秒），但能拦住所有"本地没问题线上崩了"的 bug。

### 冒烟验证是闭环的最后一环
```
idea → spec → code → test → commit → release → 冒烟验证 ← 闭环完成
                                                    ↓
                                              失败 → hotfix → 重新 release
```
没有冒烟验证，发版就是一个"我觉得没问题"的信仰行为。

## 十、全自治工作流

> 详细方案见 [`AUTONOMOUS-WORKFLOW.md`](./AUTONOMOUS-WORKFLOW.md)

### 核心理念

人只在两个时刻介入——**开头（提 idea）**和**结尾（验收结果）**。中间的 spec、review、测试、提交全部由 Agent 自驱完成。

### 12 步流水线

| 阶段 | 步骤 | 质量保障 |
|------|------|---------|
| 设计 | 调研 → 写 spec（模板化）→ self-review spec ×2 | Spec 模板强制覆盖数据流、边界 case、验收标准 |
| 测试 | 写测试（三类 case） | 正常路径 + 边界 case + 错误路径 |
| 实现 | 写代码 → self-review code ×3 | 正确性 → 健壮性 → 可维护性 |
| 验证 | 跑测试 → 视觉验证 | vitest + Playwright 截图 |
| 交付 | 文档 → commit → 冒烟验证 → 呈现 | 发版后临时目录模拟全新安装 |

### 七层工具栈

| 层次 | 工具 | 状态 | 一句话价值 |
|------|------|------|---------|
| 指令层 | Slash Commands | ⬜ 待创建 | 一句话触发完整流程 |
| 质量层 | Skills | ✅ 已装 | 专业评分框架保证每步质量 |
| 自动化层 | Hooks | ⬜ 待配置 | 不可跳过的硬性门禁 |
| 协作层 | Agent Teams | ✅ 已开启 | 多 Agent 并行替代多 terminal |
| CI 层 | Headless CLI | ✅ 可用 | 无人值守自动触发 |
| 编排层 | Agent SDK | ⬜ 待开发 | 条件分支 / 循环 / 回滚 |
| 循环层 | Ralph Wiggum | ✅ 已装 | 持续迭代直到达标 |

### 关键 insight

- **Slash Command 定义"做什么"，Skill 保证"做得好"，Hooks 确保"不能跳过"**——三层互补
- **Agent Teams 适合独立任务并行**（独立 bug、前后端分离、多方案调研），不适合同文件改动或强依赖链
- **Headless CLI 和 Hooks 互补**：Hooks 在 Agent 工作时自动触发，Headless 在无人时定期触发
- **循环 ≠ 线性**：Slash Command 走 A→B→C，Ralph Wiggum 走"做→查→不够→再做"，可组合
