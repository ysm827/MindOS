对当前改动执行结构化 code review。

使用 `code-review-quality` skill 的评分框架：
- 🔴 Blocker：必须修复才能提交
- 🟡 Major：强烈建议修复
- 🟢 Minor：建议改进
- 💡 Nit：可选优化

## Review 3 轮

### 轮 1：正确性
- 对照需求 / spec 逐条验收
- 查 `wiki/80-known-pitfalls.md`，确认没有重蹈已知陷阱
- 所有新引入的依赖版本范围是否正确？

### 轮 2：健壮性
- 外部调用（API / 文件 / 网络）是否有 try-catch？
- 用户输入是否做了验证和清洗？
- 异步操作是否有超时保护？竞态条件？
- 失败路径是否有 fallback？

### 轮 3：可维护性 + 性能
- 未使用的 import / 变量 / 函数？
- 重复代码（>3 行相同逻辑应提取）？
- 命名是否清晰一致？
- N+1 查询？大数据量 OOM 风险？
- 缓存是否三层覆盖（客户端 router cache / 服务端 revalidate / 内存 cache）？

## 输出格式

每个问题标注：
```
[🔴/🟡/🟢/💡] 文件:行号 — 问题描述 — 修复建议
```

## 自动修复

有 🔴 → 立即修复 → 重新 review 直到零 🔴
有 🟡 → 修复后说明改了什么
🟢 和 💡 → 列出但不阻塞提交
