修复 bug。描述：$ARGUMENTS

严格按 CLAUDE.md 的"Bug 处理流程"执行，不要跳步。

## 1. 复现

- 确认能稳定复现，记录复现步骤
- 如果无法复现，先分析可能的触发条件

## 2. 检查已知问题

- 搜索 `wiki/80-known-pitfalls.md`，看是否有同类问题和已知 workaround
- 搜索 `wiki/85-backlog.md`，看是否已在待办中

## 3. 写复现测试

- 先写一个能复现 bug 的测试（红灯）
- 这个测试就是修复的验收标准

## 4. 定位根因

- 不要只修表面现象，找到 **为什么** 会发生
- 画出相关的数据流 / 调用链，找到错误注入点

## 5. 全局扫描同类问题

- 搜索代码库中同样的错误模式
- 主动排查其他文件是否存在同样的 bug
- **不要只修当前文件就结束**

## 6. 修复全部

- 修当前 bug + 扫描发现的同类问题，一次性清理
- 确认复现测试变绿

## 7. 补充测试

- 除了复现测试，补充边界 case 和错误路径测试
- 过一遍 CLAUDE.md 的"边界 case 发现清单"

## 8. 自我 Code Review

- 使用 `/self-review` 的 3 轮框架
- 特别关注：修复是否引入了新问题？是否影响了其他功能？

## 9. 记录 + 提交

- 将坑记入 `wiki/80-known-pitfalls.md`（模式、根因、修复方式）
- 更新 `wiki/85-backlog.md`（打勾或新增条目）
- 跑全量测试通过后 commit + push

## 最终呈现

- Bug 根因分析
- 修复范围（改了哪些文件、为什么）
- 全局扫描结果（同类问题是否存在）
- 测试覆盖情况
