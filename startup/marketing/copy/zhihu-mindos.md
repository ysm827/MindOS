## 回答正文

同时用 3 个 AI 工具大半年了，聊聊我踩过的坑和现在的做法。

---

**先说问题：人肉上下文搬运工**

我日常用 Claude Code 写方案和架构、Cursor 写实现、Gemini CLI 跑批量脚本。

最大的问题不是工具不好用，而是**这三个工具的记忆完全隔离**。

打开 Cursor，要把项目背景重新讲一遍。切到 Claude Code，再讲一遍。Gemini CLI 连上一次对话的记忆都没有。

我粗算过这个时间成本：每次切工具交代背景大概 3-5 分钟，每天切 5-8 次，**一天花 15-30 分钟在"给 AI 交代它本来应该知道的事情"上**。一周就是 2-3 小时。

更痛的是纠正不会被记住。我告诉 Claude "错误码用枚举不用字符串"，当次有效，下次开新对话又用字符串。

---

**你可能会说：不是有 .cursorrules 和 CLAUDE.md 吗？**

对，我也用过。但这些是**单工具内的方案**：

| 方案 | 覆盖范围 | 问题 |
|------|---------|------|
| `.cursorrules` | 只有 Cursor | Claude Code 和 Gemini 读不到 |
| `CLAUDE.md` | 只有 Claude Code | Cursor 和 Gemini 读不到 |
| 共享 `docs/` 文件夹 | 需要手动 @ 或粘贴 | 每次还是要人工搬运 |

如果你只用一个 AI 工具，`.cursorrules` 完全够用。但同时用 3 个以上，你就是在三个地方各维护一份上下文，还得保持它们一致。

---

**我现在的做法：一个共享知识层，所有工具直连**

核心思路：**把项目上下文统一写在一个地方，通过 MCP 协议让所有工具自己来读。**

MCP（Model Context Protocol）目前支持的工具：
- ✅ Claude Code、Cursor、Cline、Zed、Windsurf、VS Code (Copilot)
- ❌ ChatGPT 网页版、Gemini 网页版（不支持 MCP，但 Gemini CLI 支持）

具体结构长这样：

```
my-mind/
├── Profile/          # 你是谁、技术栈、工作偏好
│   └── me.md
├── Projects/         # 每个项目的架构决策、技术选型
│   ├── project-a/
│   └── project-b/
├── Workflows/        # SOP、代码规范、review checklist
│   ├── coding-standards.md
│   └── deploy-checklist.md
├── Preferences/      # 命名规范、错误处理偏好等
│   └── conventions.md
└── INSTRUCTION.md    # 所有 Agent 必须遵守的规则
```

每个 AI 工具连上后，会自动读到这些文件。你在 `Preferences/conventions.md` 里写了"错误码用枚举"，三个工具全都知道。

---

**手动管 vs MCP 直连，差别在哪**

| 维度 | 手动管理（docs/ + 各工具 rules） | MCP 直连 |
|------|-------------------------------|---------|
| 维护成本 | 每个工具各一份，要保持同步 | 一处修改，全局生效 |
| 上下文传递 | 要手动 @ 或粘贴 | 自动读取，零操作 |
| 纠正沉淀 | 纠正只在当次对话有效 | 写回知识库，永久生效 |
| 新增工具 | 要新建一份 rules 文件 | 配一次 token 即可 |

如果你只用 1-2 个工具，手动管够了。如果 3 个以上，MCP 直连的效率差距会越来越大。

---

**效果**

用了两个月，三个变化：

1. **交代背景的时间从每天 20 分钟降到接近零。** 不是因为我记忆力变好了，是工具自己去读了。
2. **三个工具的输出一致性明显提升。** 不再出现 Claude 以为用方案 A 而 Cursor 以为用方案 B 的情况。
3. **知识库在复利增长。** 每次纠正 AI 都写回去，下次所有工具都变准。两个月积累了 50+ 条规则，现在 AI 的输出质量比最初好很多。

---

**工具和局限性**

我用的是自己做的开源项目 MindOS，所以我的体验有创始人滤镜，仅供参考。

坦诚说几个局限：
- MCP 协议目前 OpenAI 原生不支持，如果你的主力工具是 ChatGPT 网页版，这个方案对你帮助有限
- 初始的知识库搭建需要花 1-2 小时把现有项目信息整理进去（之后是增量维护）
- 如果你只用一个 AI 工具，这个方案的价值不大，`.cursorrules` 就够了

但如果你和我一样同时用 3 个以上 AI 工具，并且受够了重复交代背景——这个思路值得一试。

具体怎么搭、结构怎么设计，评论区可以细聊。

## 优化要点总结

| 改动 | 原因 |
|------|------|
| 去掉"谢邀" | 2026 年已是过时老梗 |
| 加时间成本推导 | "每次 3-5 分钟 × 每天 5-8 次"比"20-30 分钟"更可信 |
| 加 MCP 支持/不支持列表 | 诚实说明适用范围，减少杠精攻击面 |
| 加手动管理 vs MCP 对比表 | 回应"我用 .cursorrules 也行"的质疑 |
| 加真实目录结构示例 | 从"怎么做"到"照着做"，提升收藏价值 |
| 结尾坦诚说局限性 | "创始人滤镜"自嘲降低软文感，知乎用户尊重诚实 |
| 去掉"这个回答核心不是推荐工具" | 此地无银三百两，不如直接坦诚 |
