# MindOS 商业计划书

**让你的每次思考都在增值——不只是这一次对话。**

---

### 核心术语

| 术语 | 定义 | 复利类比 |
|------|------|---------|
| **判断（Judgment）** | 用户对某个决策的选择与偏好——比如"API 错误一律用枚举而非字符串" | 复利的"本金" |
| **沉淀（Crystallize）** | 判断转化为可复用的 Skill（执行手册）或 Instruction（治理规则） | 复利的"利率" |
| **复用（Reuse）** | 所有 Agent 无需重复提问，直接读取已沉淀的判断并执行 | 复利的"收益" |

## 一、执行摘要

### 公司概述

AI 工具正在裂变——写代码的、做设计的、搜信息的、管项目的，每个场景都有专门的 Agent。用户不是选择用多个 AI，是不得不用多个。但每个 AI 都是独立的孤岛：切工具重新讲背景，纠正了明天还犯，偏好设了换一个又要重设。

问题不只是繁琐。用户在对话中做出的上百个判断——架构决策、代码规范、设计偏好——是他最高密度的认知资产。但对话关掉的那一刻，全部消失了。AI 的进步是复利的，每一代建立在上一代之上。用户的进步是线性的，每次对话关掉都从零开始。

**MindOS 是你和所有 AI 之间的判断沉淀层。** 你写下判断，所有 AI 遵守。你纠正 AI，纠正变成规则。飞轮转起来后，AI 越用越准（Agent 复用规则），人的思维也越来越清晰（写显性规则的过程锐化了判断力）。这就是认知复利——不只是效率工具，是人和 AI 在同一个系统里螺旋上升。

**品类：** 认知外化平台。不是笔记工具（Obsidian 的读者是人），不是 AI 记忆（MemOS 是黑箱），不是 Agent 编排（CrewAI 调度 Agent 做什么）。MindOS 告诉 Agent 你是谁。

> 完整 BMC 分析详见 `bmc-mindos.zh.md`。

### 产品状态

- **阶段：** P1.5（核心功能已交付，进入用户验证冲刺）
- **已实现：** GUI 工作台 + MCP Server + 一键导入 + ACP/A2A 双协议 + Workflow 编排 + Skills 系统 + Echo 回响 + 桌面端（三平台）+ CLI + 新手引导
- **发布：** npm 已发布，GitHub 开源
- **用户：** 内测群约 30 人。尚无日活数据和付费意愿验证
- **当前最高优先级：** 不是写新功能，是见用户——4 周验证冲刺

### 飞轮

```
① 痛点入口    "别重复交代"           → 安装 MindOS
② 判断积累    纠正沉淀为规则          → 规则越来越多
③ AI 变准     Agent 复用规则          → 用户更依赖
④ 人变清晰    写规则锐化思维          → 思维变清晰（锁定）
```

飞轮最精妙的一环是第四步：用户的思维方式被改变了。迁移成本不是数据级别的（Markdown 随时带走），是认知级别的。最致命的缺口在第二步：用户是否愿意把判断写成规则？经验编译技术（纠正 3 次自动生成规则）是弥补这个缺口的关键，在 P2 阶段优先投入。

### 三层问题

1. **使用割裂** — 每个 AI 是孤岛，你在孤岛之间来回搬运上下文
2. **经验在蒸发** — 对话中的判断关掉就没了，没有系统在收集
3. **成长不复合** — 你在变强，但进步散落在 100 个对话里，从未复合

> **证据状态：** 以上痛点来自创始人自身经历和内测群口头反馈，尚未经过系统化用户验证。这是当前最大的已知缺口。

### 解决方案

- **判断沉淀** — 纠正和偏好自动变成规则，不再丢失
- **跨 Agent 复用** — MCP + ACP 双协议，所有 Agent 读同一套规则
- **人可治理** — 纯 Markdown，你随时审查、修正、删除
- **5 分钟 Aha** — 安装后导入已有规则文件，Agent 立刻遵守

### 三层护城河

| 层 | 壁垒 | 机制 |
|----|------|------|
| 方向壁垒 | 大厂做 AI 的记忆（黑箱、锁定），MindOS 做人的记忆（透明、自由切换）。方向相反 | 结构性 |
| 算法壁垒 | 经验编译、认知镜像、记忆代谢、上下文路由——本地优先+纯文本+单用户约束下的原创算法 | 研究级 |
| 数据壁垒 | 每个用户的判断规则独一无二，用越久迁移成本越高 | 时间积累 |

## 二、问题与市场机会

### 问题

| 痛点 | 描述 | 频率 | 证据来源 |
|------|------|------|---------|
| 上下文割裂 | 切换 Agent 靠人工搬运上下文，每次对话重新交代背景 | 每天多次（假设） | 创始人经验 + 内测群反馈（N≈10） |
| 治理黑箱 | Agent 记了什么、记对没有，用户无法审查和修正 | 每次使用（假设） | 创始人经验，待用户观察验证 |
| 判断断流 | 对话中的纠正和判断关掉就丢，无法沉淀复用 | 每次对话（假设） | 创始人经验，待用户观察验证 |

> **下一步验证计划：** 找 10 个非朋友用户做深度访谈，量化"反复讲背景"的时间成本（目标：证明平均每天 >15 分钟用于重复交代上下文）。

### 真实竞争对手

不是某个单一产品，而是用户当前"多工具拼接"的现状工作流：在 Claude / Cursor / Gemini CLI / 文档 / 笔记 / 本地文件夹之间来回切换，靠人手搬运上下文。

### Why Now

Vibe Coding 时代，"做出来"不再稀缺，"想清楚"才稀缺。AI 已经能写代码、做设计、跑流程，但所有 AI 工具都只负责输出，不负责沉淀。MCP 协议的标准化（2025-2026）使得跨 Agent 共享上下文第一次成为可能。

### 市场趋势

1. **多 Agent 工作流成为常态** — 开发者平均使用 3+ AI Agent
2. **MCP 协议被越来越多工具采纳** — 已支持：Claude Code、Cursor、Cline、Zed、Windsurf、VS Code（Copilot）等；尚未采纳：OpenAI 原生、Google Gemini 原生
3. **本地优先回潮** — 数据隐私意识增强，开发者偏好本地控制
4. **AI 记忆赛道升温** — OpenAI Memory、SuperLocalMemory 等产品出现，验证了市场需求

### 目标用户

**唯一核心 Persona（P1 ~ P2 阶段只服务这一种人）：**

每天打开 3 个以上 AI 对话窗口的独立开发者——早上用 Claude Code 写后端、下午用 Cursor 改 UI、晚上用 Gemini CLI 跑脚本。他最烦的事不是 AI 不够聪明，而是每次开新对话都要重新交代"我的项目用 TypeScript、API 错误用枚举、部署在 Fly.io"。

**判断依据：** 创始人本人就是这个用户。验证方式不是写报告，而是在 X/即刻发一条"你每天同时用几个 AI Agent？"看回复数和共鸣度。如果这条推文没有 engagement，这个品类不成立。

**暂不追求的用户（P3 再考虑）：**
- AI-native 小团队（3-15 人）— 需求真实但获客成本高，需要先验证个人用户 PMF
- 系统性思考者 / Markdown 爱好者 — 会把 MindOS 当 Obsidian 用，不是核心价值

## 三、产品与解决方案

### 产品定义

MindOS 不是"给笔记加 AI"（Obsidian 的读者是人），也不是"给 Agent 加记忆"（MemOS 的用户是开发者）。MindOS 是**人和 AI 之间的治理层**——让你的判断持续沉淀、被所有 Agent 复用、由你审计和修正。

**Obsidian 是你的第二大脑，MindOS 是你和 AI 的共享大脑。**

### 与 Obsidian 的本质区别（必须在产品体验中传达）

用户第一眼看到编辑器界面，必然会想"这不就是另一个 Obsidian 吗？"。这说明产品的第一印象设计有问题。

**解决方案：让用户第一眼看到的不是编辑器，而是 Agent 连接面板。**

| 维度 | Obsidian | MindOS |
|------|---------|--------|
| 心智模型 | 人写笔记 | 人和 Agent 共治一个知识库 |
| 首屏 | 文件树 + 编辑器 | Agent 连接状态 + "你有 3 个 Agent 连着你的知识库" |
| 核心交互 | 写和读 | 写、读、**被 Agent 读** |
| 数据流向 | 人 → 文件 → 人 | 人 → 文件 → Agent → 人 |

**具体改法：**
- Landing page 不展示编辑器截图，展示 Agent 连接图
- 首次打开不进编辑器，进 Agent 连接向导
- Dashboard 首屏展示"今天 Agent 读取了你的知识库 47 次"

> **Obsidian 的 MCP 插件只是把数据暴露给 Agent，不是把 Agent 生态连接成一个系统。** Obsidian 没有 Instruction（Agent 治理规则）、没有 Skill（判断沉淀），没有多 Agent 上下文同步。但如果用户感知不到这个区别，差异化就不存在。

### 5 分钟 Wow Moment（P1 最高优先级）

**认知复利是留存故事，不是获客故事。获客故事是"AI 终于认识你了"。**

用户安装 MindOS 后 5 分钟内必须经历这个瞬间：第一次在 Claude Code 里输入 `@mindos`，Agent 说出了他昨天记的技术决策——"卧槽，AI 居然知道我昨天想的事"。

**实现路径：**
1. `mindos onboard` 时自动生成一份 Profile（从 GitHub / 系统信息抽取）
2. 连接第一个 Agent 后自动触发一次对话，Agent 主动引用 Profile 内容
3. 用户在 30 秒内感受到"这个 Agent 认识我"

**如果 Wow Moment 来得太晚（>10 分钟），留存曲线会在 Day 2-7 断崖。首次体验不是 P1.5 的事，是 P1 最重要的事。**

### 产品策略：双层楔子

| 层级 | 价值 | 对应功能 | 用户感知时间 |
|------|------|---------|------------|
| **获客层** | "AI 终于认识你了"——第一次连接就有用 | MCP Server、CLI onboard、自动 Profile 生成 | 5 分钟内 |
| **留存层** | 判断沉淀——纠正和偏好自动沉淀为 Skill | Skills、经验回流、Instruction 规则 | 第 1-2 周 |
| **差异化层** | 认知复利——用的越久 AI 越准 | 知识库演进、方法论沉淀、跨 Agent 上下文同步 | 第 1 个月后 |

**一句话：先卖"AI 终于认识你了"（5 分钟），再卖"再也不用反复讲背景"（第 1 周），最后用户自己发现"我的判断开始复利"（第 1 个月）。**

### 核心功能

**面向人类：**
- GUI 工作台：双模式编辑器（富文本 + Markdown）、AI 对话（⌘/）、全局搜索（⌘K）、Wiki Graph
- 11 个渲染器插件：TODO Board、CSV Views、Wiki Graph、Timeline、Agent Inspector 等

**面向 Agent：**
- MCP Server：stdio + HTTP 双传输，兼容 Claude Code、Cursor、Gemini CLI、Windsurf 等 10+ Agent
- ACP 协议适配：Agent Communication Protocol 接入层，让 MindOS 被 ACP 生态的 Agent 发现和调用（轻量实现，2-3 周工作量，卡位价值大）
- 多 Agent 上下文同步：不是 CrewAI 式的任务编排引擎，而是确保所有 Agent 读到同一套 Instruction/Skill/Profile。场景：Agent A（Claude Code）写后端、Agent B（Cursor）写前端，两者都读到"API 错误一律用枚举"的 Instruction，行为一致
- MindOS Skills：结构感知路由，对话中的判断可沉淀为可复用 Skill
- 结构化模板：预设 Profile、Workflows、Projects 等目录结构

**基础设施：**
- CLI 工具链：onboard / start / sync / token，30 秒完成安装到 Agent 连通
- Git 自动同步：自动 commit/push/pull，冲突保留 `.sync-conflict`
- 安全：Bearer Token、路径沙箱、INSTRUCTION.md 写保护、原子写入
- 桌面端：macOS / Windows / Linux 原生应用

### 核心概念

| 概念 | 定义 | 对应术语 |
|------|------|---------|
| **Space** | 按用户思维方式组织的知识分区 | 结构层 |
| **Instruction** | 所有 Agent 都遵守的治理规则 | 控制层 |
| **Skill** | 沉淀的判断转化为可复用的执行手册 | 执行层 |

三者形成：**结构层（Space）→ 控制层（Instruction）→ 执行层（Skill）**

### 产品路线图（验证优先，功能其次）

| 阶段 | 时间 | 重点 | Kill Criteria |
|------|------|------|--------------|
| **P0.5** | **Month 1**（当前） | **4 周验证冲刺**：5 人深度访谈 + Waitlist 价格页 + 3 人 1v1 onboarding + 基础埋点 | 4 周后无正面信号 → 考虑 pivot 为 Obsidian/Cursor 插件 |
| P1 | Month 2-6 | 核心体验：5 分钟 Wow Moment + MCP 连接打磨 + 使用数据采集 | Month 6 日活 < 30 人 → 认真考虑这是否是独立产品 |
| P1.5 | Month 6-9 | ACP 协议适配（轻量卡位）+ 用户反馈驱动的功能迭代 | — |
| P2 | Month 9-15 | RAG / 语义搜索 + Pro 订阅上线（前提：MAU 500+ 且 7 日留存 >30%） | Pro 层 Month 12 付费 < 20 人 → 重新评估定价和功能 |

**关键调整：**
1. 新增 P0.5 阶段——在写任何新功能之前先验证需求
2. P1 最高优先级从"功能完整"改为"5 分钟 Wow Moment"
3. 每个阶段都有 Kill Criteria，不达标就止损或 pivot

### Plan B：如果独立产品不成立

如果 Month 9 时日活 < 30 人，MindOS 可能不应该是独立产品，而应该成为现有生态的插件：
- **方案 A：Obsidian 插件** — 利用 Obsidian 66K stars 的用户基础，做"Obsidian + Agent 连接层"
- **方案 B：Cursor/VS Code 扩展** — 作为 IDE 内置的知识库面板，获客成本接近零
- **代价：** 天花板更低，无法独立定价，但验证了核心价值（跨 Agent 上下文共享）

### 不做什么

- 不做通用笔记工具 — 不在编辑器层和 Notion/Obsidian 竞争
- 不做自建富文本格式 — 依赖纯 Markdown/CSV，保证可迁移
- 不做通用 Agent 编排引擎 — 不和 CrewAI/AutoGen/LangGraph 竞争，只做"上下文同步"
- 不做实时多人协同编辑 — 支持团队异步共享
- 不做 Agent 训练/微调 — 只做知识存储和工具调用
- **不在 PMF 验证前开发 Pro 层功能** — 先证明免费层有人用，再做付费功能

## 四、商业模式

### 收入模型

| 层级 | 价格 | 包含 | 为什么值钱 |
|------|------|------|----------|
| Free | $0 | 本地 CLI + MCP/ACP 协议适配 + GUI + Git 同步 + 基础渲染器 + 基础多 Agent 编排 | 用户增长引擎，永不锁定数据 |
| Pro | ~$8/月 | Knowledge Health + Agent Memory 审计 + 高级 RAG + Cloud Sync + AI 智能编排 + Context 智能路由 + 编排效果分析 | AI 计算增值——当知识库 >500 文件时，语义搜索和健康度分析显著提升 Agent 质量；多 Agent 场景下智能编排大幅降低人工协调成本 |
| Team | ~$15/人/月 | 多 Mind + Agent Governance + 审计日志 + SSO + 团队编排治理 + 编排模板市场 | 团队级治理——合规底线 + 多人协作 + 团队编排最佳实践共享 |

### 开源 vs 闭源划线策略（Open Core 模型）

**核心原则：开源"连接"，闭源"智能"。协议层是分发渠道，AI 计算层是利润中心。**

**Open Core 的壁垒不在代码本身，在三件事：**
1. **集成深度** — 不是通用编排，是知识库感知的上下文同步。社区可以抄代码，但抄不走数据模型和用户习惯
2. **运维成本** — 自己搭 embedding + 向量数据库 + LLM 推理全链路，大多数开发者宁愿付 $8/月
3. **持续迭代** — 社区 fork 只是某个时间点的快照，Pro 层的 AI 功能迭代速度由核心团队决定

```
┌──────────────────────────────────────────────────────────────┐
│  永远开源（Free）— 获客引擎                                   │
│  本地存储、编辑、MCP/ACP 协议适配、CLI、基础 GUI、             │
│  Skills/Instruction 框架、基础渲染器（Markdown/CSV/TODO）、    │
│  Git 同步、桌面端壳、Agent 注册/发现/连接、                    │
│  多 Agent 上下文同步（所有 Agent 读同一套 Instruction/Skill）   │
│  → 用户掌控数据，零锁定；越多 Agent 接入，网络效应越强          │
├──────────────────────────────────────────────────────────────┤
│  闭源付费（Pro ~$8/月）— AI 增值层                            │
│  语义搜索/RAG、Knowledge Health 评分、自动 Skill 提炼、        │
│  Agent 审计分析（趋势/异常检测）、                              │
│  Context 智能路由（AI 判断哪些上下文对哪个 Agent 有用）、       │
│  冲突检测与仲裁（多 Agent 同改一文件时 AI 仲裁）、              │
│  Cloud Sync + 多设备、高级渲染器（Knowledge Graph/Analytics）   │
│  → 需要 LLM 推理 + embedding + 服务端，社区难以复制             │
├──────────────────────────────────────────────────────────────┤
│  闭源付费（Team ~$15/人/月）— 服务端协调层                     │
│  多人共享 Mind、权限管理、合规级审计日志、SSO/SAML、            │
│  团队级 Instruction 治理（审批流/权限控制）                     │
│  → 需要中心化服务端，本地做不了                                │
└──────────────────────────────────────────────────────────────┘
```

### MindOS 的差异化

1. **不只是记忆，而是可治理的上下文层** — Agent 读写留痕，人类可审计
2. **判断沉淀与复用** — 纠正沉淀为 Skill/SOP，竞品都没有这个能力
3. **人机共享心智** — 同一套知识同时服务人类阅读和 Agent 执行
4. **品类定义** — Personal Context OS，不是笔记工具，不是 Agent 记忆

### 重点竞品：MemOS

**MemOS**（[MemTensor/MemOS](https://github.com/MemTensor/MemOS)，7.8K stars，Apache 2.0）是目前最接近的竞品。它自称"AI Memory OS"，为 LLM 和 Agent 系统提供统一的记忆层。支持三种记忆类型（明文记忆、KV cache 激活记忆、参数记忆/LoRA），有学术论文支撑（2025.07），且已发布 Claude Code 插件。v2.0.10，2026 年 3 月仍在活跃迭代。

**MemOS vs MindOS 关键区别：**

| 维度 | MindOS | MemOS |
|------|--------|-------|
| 核心用户 | 终端用户（开发者/创始人） | 开发者/研究者（Python SDK） |
| 交互方式 | GUI 工作台 + CLI + MCP | Python API + REST API |
| 记忆形态 | Markdown 纯文本（人可读写） | 结构化数据（图数据库 Neo4j/向量） |
| 人可治理 | GUI 审计面板，人类直接编辑 | API 级别，需编程操作 |
| 判断沉淀 | Skill/SOP 经验回流 | 无（专注记忆存储和检索） |
| 学术背景 | 无 | 有论文（MemOS: A Memory OS for AI System） |
| 生态位 | 面向终端用户的"思考的地方" | 面向开发者的"记忆基础设施" |

**结论：** MemOS 是基础设施层竞品，MindOS 是应用层产品。两者可能共存甚至互补——MindOS 后端 RAG 未来可接入 MemOS 作为记忆引擎。但如果 MemOS 向上做 GUI，或 MindOS 不能在用户体验上拉开差距，两者会直接竞争。

### 最大威胁：Agent 厂商自建

**风险：** Anthropic/OpenAI 可能在 Agent 层原生实现跨 Agent 上下文共享。他们有协议、有用户、有数据。

**应对策略：**
- MindOS 的核心价值在"人可治理 + 判断沉淀 + 跨生态编排"，不在"记忆存储"或单一协议本身——厂商做记忆，MindOS 做治理层和编排层
- 保持跨生态兼容——不绑定单一 Agent 厂商，同时支持 MCP + ACP
- 如果 MCP/ACP 未成为行业标准，Plan B 是支持多协议适配（MCP + ACP + OpenAI function calling + 自定义 API）
- 开源协议适配层和基础编排，抢占"标准参考实现"身份，避免被竞品的开源替代绕过

### 护城河

- **网络效应：** 知识库越丰富 → Agent 越好用 → 用户越不想走；接入的协议越多（MCP + ACP） → 能同步的 Agent 越多 → 越多用户选择 MindOS 作为中枢
- **数据沉淀：** 用户的判断、偏好、方法论沉淀在本地，迁移成本随时间增长
- **知识库感知：** 通用编排框架（LangChain/CrewAI）不了解用户的 Instruction 和 Skill 体系。MindOS 的上下文同步基于用户的认知资产结构，而非通用的 key-value 记忆
- **标准先发：** MCP + ACP 双协议的 Personal Context 层目前无标准参考实现
- **生态锁定：** Skill 生态建立后，社区贡献形成正向循环

> **诚实评估：** 以上护城河在当前阶段都是理论性的。真正的护城河在用户量 >1000 后才开始显现。目前最重要的不是护城河，是验证产品是否有人需要。

1. 用户正常工作，判断和偏好持续沉淀
2. 任何新 Agent 接入，零配置就能读到完整的认知资产
3. 用户的感受是："我终于不是在伺候 AI，而是 AI 在伺候我积累下来的判断。"

### 长期使命

让每个人在 AI 时代都能积累属于自己的认知资产——判断不再随对话消散，而是持续增值、被所有工具复用。
