# MindOS 技术支撑

*最后更新：2026-03-31 | 技术细节详见 `wiki/03-technical-pillars.md`*

---

## 核心论点

MindOS 的战略愿景（认知复利 → 人机共演化 → 五层积累 → 双杠杆）不是空中楼阁。每一层愿景都有对应的技术支柱支撑。

---

## 愿景到技术的映射

```
战略愿景                              技术支柱
────────────────────                 ────────────────────
层次 1: 记住                    ←── MCP Server + Markdown 知识库
层次 2: 结构化                  ←── 经验编译（交互→SOP 自动提取）
层次 3: 反思（人变强）          ←── 认知镜像（模式识别+反哺）
层次 4: 方法论                  ←── 记忆代谢（知识库自组织）
层次 5: 代理                    ←── 群体调度（多 Agent 治理）+ 品味代理

认知复利飞轮                    ←── 四根技术支柱的协同运转
人机共演化                      ←── 认知镜像（反哺人）+ 经验编译（AI 变准）
双杠杆（Vibe Coding + IP）      ←── 上下文路由（手）+ 方法论提炼（心）
```

## 四根技术支柱

### Pillar 1：群体调度 — 多 Agent 有序共存

**解决什么：** 用户同时接入 5-10 个 Agent，并发读写同一个知识库，谁来维持秩序？

**当前状态：** MCP Server 已支持多 Agent 同时连接（HTTP + stdio），Bearer Token 提供身份识别。

**待突破：**
- 任务感知的上下文路由（重构代码的拿完整架构，拼写检查的只拿当前段落）
- 并发冲突解决（两个 Agent 同时改一个文件）
- 细粒度权限（Claude Code 可写代码区但只读 Profile）

**支撑哪层愿景：** 层次 5（代理）、P3（Agent Governance）

---

### Pillar 2：经验编译 — 交互自动变成规则

**解决什么：** 用户纠正了 Agent 的错误，这个纠正只活在当前对话里。怎么自动变成 SOP？

**当前状态：** Skills 的 SKILL.md + experience.md 已形成半结构化机制，但完全依赖人工。

**待突破：**
- 反向编译：Agent 出错 → 人修正 → 系统自动提取 diff → 结构化为 SOP 补丁
- 增量演化：在已有 SOP 上局部修改，不全量重写

**支撑哪层愿景：** 层次 2（结构化）、层次 3（反思）。这是"认知复利"的技术核心——让"改"信号自动变成可复用的规则。

---

### Pillar 3：记忆代谢 — 知识库自动保鲜

**解决什么：** 知识库只增不减，Agent 越用越迟钝（上下文污染）。

**当前状态：** Wiki Graph 提供文件间链接拓扑，Agent Inspector 记录调用日志。

**待突破：**
- 记忆分层：工作记忆 → 短期记忆 → 长期记忆，自动升降级
- 活跃度感知：高频节点预热，低频节点归档
- 矛盾检测：同一主题不同文件出现矛盾时主动标记

**支撑哪层愿景：** 层次 4（方法论）。知识库能自组织，方法论才能从中涌现。

---

### Pillar 4：认知镜像 — 深度理解用户

**解决什么：** 通用 Agent 对所有用户一视同仁。怎么让 Agent "懂你"？

**当前状态：** Profile 目录存储用户身份/偏好，Agent Inspector 记录交互轨迹。

**待突破：**
- 偏好推断：从编辑行为（用户修改了 AI 写的内容）中提取隐性偏好
- 心智预设注入：新 Agent 连上后立刻"认识"用户（`_meta.user_profile`）
- 演化追踪：记录用户认知随时间的变化（"三个月前你是怎么想这个问题的"）

**支撑哪层愿景：** 层次 3（反思）——AI 反哺人成长的技术基础。也是"人机共演化"的关键：AI 不只执行，还能照见人的盲区。

---

## 四根支柱的结构关系

```
Pillar 1 (群体调度)  →  Pillar 2 (经验编译)  →  Pillar 3 (记忆代谢)  →  Pillar 4 (认知镜像)
   环境：谁能进来        产生：经验变知识         维护：知识不腐化         涌现：深度理解用户
```

类比操作系统启动顺序：
- 内核（进程管理、权限）→ 编译器（源码→产物）→ 文件系统（存储、清理）→ 用户态（个性化）

工程上可并行推进，但存在产品逻辑上的前置依赖：没有有序的多 Agent 环境，经验编译的数据来源不可靠；没有前三者的积累，认知镜像无从涌现。

## "做改看弃"信号与技术支柱的对应

| 信号 | 由哪个 Pillar 处理 | 怎么处理 |
|------|-------------------|---------|
| **做**（用户操作轨迹） | Pillar 4（认知镜像） | 从操作轨迹推断隐性偏好和 SOP |
| **改**（用户修改 AI 输出） | Pillar 2（经验编译） | 提取 diff 自动结构化为规则 |
| **看**（注意力停留） | Pillar 3（记忆代谢） | 更新知识节点活跃度权重 |
| **弃**（中断/忽略） | Pillar 1（群体调度） | 划定 Agent 行为边界，剪枝错误路径 |

## 技术护城河

**为什么这些技术不容易被复制：**

1. **数据飞轮**：每个用户的"做改看弃"数据都是独一无二的。积累越多，认知镜像越准，越难被替代。
2. **本地优先约束**：在没有云端大数据的条件下做个性化——这是一个独特的技术挑战，大厂的协同过滤方案用不了。
3. **Markdown 作为执行层**：MindOS 证明了纯 Markdown 可以被 Agent 可靠执行，不需要 JSON/DSL 中间层。这个设计决策降低了用户门槛但提高了技术难度（上下文路由、碎片折叠都要在纯文本上做）。
4. **经验编译**：从非结构化对话中自动提取结构化 SOP——这是一个学术级别的研究问题，不是简单的工程实现。
